## Dictionary Coding: the LZ Family and the Codecs That Run the World

An **entropy coder** (Huffman, arithmetic, ANS) shortens a stream symbol-by-symbol according to how surprising each symbol is. But it has no memory of *patterns*: to it, the string `the the the` is just nine letters and two spaces, each priced by its marginal frequency. Yet that string is wildly redundant — it is one word repeated. **Dictionary coding** captures exactly this kind of redundancy: instead of re-emitting a chunk of data you have already seen, you emit a short *reference* back to the earlier occurrence. This is the single most important idea behind the compressors that move essentially all data on the internet.

### LZ77 and LZ78: two ways to remember the past (1977–1978)

The foundational insight is due to **Abraham Lempel and Jacob Ziv**, two researchers at the Technion in Haifa. In their 1977 paper (*IEEE Transactions on Information Theory*, May 1977) they introduced what we now call **LZ77** (sometimes "LZ1"). The encoder keeps a **sliding window** over the most recently seen bytes. At each position it searches the window for the longest run of upcoming bytes that already appeared, and emits a triple: **(distance, length, next literal)** — "go back `distance` bytes, copy `length` of them, then append this literal." A match is thus a *back-reference*, not the data itself. On `abcabcabc`, after emitting `abc` literally, the encoder can copy the rest with a single (distance=3, length=6) reference. The window bounds memory and gives the scheme its self-adapting flavor: the "dictionary" is simply the recent text, so it tracks local statistics for free.

Their 1978 follow-up, **LZ78** ("LZ2"), takes a different tack: it builds an *explicit* dictionary of previously seen phrases, indexed by number, and emits **(index, next-symbol)** pairs that grow the dictionary as they go. LZ77 references positions in a window; LZ78 references entries in a table. Crucially, Lempel and Ziv proved these schemes are **asymptotically optimal**: for a stationary ergodic source, the compression rate converges to the source's entropy rate, *with no prior knowledge of the statistics*. This was the theoretical bombshell — universal compression that learns the source as it reads it.

### LZW, and the patent that broke GIF (1984–2003)

In 1984 **Terry Welch**, at Sperry, published a streamlined LZ78 variant, **LZW** (Lempel-Ziv-Welch), that emits only dictionary indices (the "next symbol" is folded into the table growth). It was simple, fast, and hardware-friendly — and it was patented (US 4,558,302, issued December 1985). LZW became the compressor inside **GIF** (CompuServe, 1987) and Unix `compress`. Then, on December 24, 1994, **Unisys** (Sperry's successor) announced it would enforce the patent and demand royalties from software using GIF. The backlash birthed **PNG** (1996), a deliberately royalty-free format using the unencumbered DEFLATE instead. The patents expired around 2003–2004. The episode is the canonical cautionary tale of how intellectual property, not engineering merit, can decide which formats survive.

### DEFLATE: the workhorse that quietly won (1991–1996)

The combination that conquered the world is **DEFLATE**, designed by **Phil Katz** for PKZIP 2 around 1991 and specified in **RFC 1951** (1996). DEFLATE is beautifully pragmatic: run **LZ77** to turn the data into a stream of literals and back-references, then feed *that* stream through **Huffman coding** so the common literals and match-lengths get short codes. It is dictionary coding *and* entropy coding stacked — the two ideas are complementary, not competing. Katz let the patent lapse into practical irrelevance, and DEFLATE became free to use. The result is ubiquity almost without parallel: it is the engine of **zlib**, **gzip**, **PNG**, the **ZIP** format, and HTTP's `Content-Encoding: gzip`. For thirty years, "compression" on the web has mostly *been* DEFLATE.

### The modern frontier: ratio vs. speed vs. memory

DEFLATE's limits are structural: a 32 KB window (it can't reference far-away repeats) and Huffman coding (which wastes up to ~1 bit per symbol because code lengths are integers). The post-2000 lineage attacks these along a **four-way trade-off**: compression ratio, encode speed, decode speed, and memory. You cannot maximize all four; codecs pick a corner.

- **LZMA** (Igor Pavlov, ~1998, the core of **7-Zip**) goes for *ratio*. It pairs a huge dictionary (up to gigabytes) with a **range coder** — an arithmetic-style entropy coder using context models — so it squeezes much harder than DEFLATE, at the cost of slow, memory-hungry encoding. This is the `xz` you wait on to compress a kernel tarball.
- **bzip2** (Julian Seward, 1996) is a different animal entirely: it uses the **Burrows-Wheeler Transform**, which permutes the data to cluster similar contexts before entropy coding. It often beats DEFLATE on ratio but is slow both ways, and has been largely superseded.
- **LZ4** (Yann Collet, 2011) and **Snappy** (Google, 2011) go for raw *speed*: a fast LZ77 match-finder, **no** entropy-coding stage at all, decoding at gigabytes per second. You accept a worse ratio in exchange for compression that is nearly free — ideal for databases, RPC, and live data paths.
- **Brotli** (Jyrki Alakuijala and Zoltán Szabadka, Google, open-sourced September 2015, **RFC 7932**, 2016) targets *the web specifically*. It is LZ77 + Huffman + context modeling, but ships with a **120 KB static dictionary** of common web tokens (`<html>`, `function`, `</div>`). For HTML/CSS/JS at high levels it beats gzip by 15–25%, which is why browsers added `Content-Encoding: br`. It also powers **WOFF2** web fonts.

### Why Zstandard won

**Zstandard / zstd** (Yann Collet at Facebook/Meta, v1.0 released August 31, 2016; **RFC 8878**) is the codec that has, since the late 2010s, displaced gzip across infrastructure. Its appeal is that it refuses to pick *one* corner of the trade-off — it gives you a **dial**, levels from −7 (faster than DEFLATE) to 22 (near-LZMA ratio), all in one decoder. Several design choices make this possible. Its entropy stage uses **FSE** (Finite State Entropy), Collet's implementation of **Asymmetric Numeral Systems** — Jarek Duda's 2009 family of coders that achieve arithmetic-coding-level optimality at table-lookup speeds, dodging the integer-bit waste of Huffman. It supports **trained dictionaries**: feed zstd thousands of small similar records (log lines, JSON blobs) and it builds a custom dictionary, transforming the otherwise-hopeless problem of compressing tiny payloads. And it decodes fast regardless of compression level, which matters because data is compressed once and read many times.

Adoption tells the story: zstd has been in the **Linux kernel since 4.14** (November 2017) for **btrfs**, squashfs, and module compression; it is a default in Btrfs and a transport codec across Meta, AWS, and others; and in 2024 both **Chrome 123** (March) and **Firefox 126** (May) shipped HTTP `Content-Encoding: zstd`. The reference implementation continues to evolve — **v1.5.7** (February 2025) added hooks letting Intel **QAT** hardware accelerate the LZ match-finding stage. The honest summary of the field as of 2026: pick **LZ4/Snappy** when CPU is the bottleneck, **xz/LZMA** when you need the absolute smallest file and can wait, **Brotli** for static web assets, and **zstd** for almost everything else — which is precisely why it has become the new default.
